# 通过 OAuth 方式获取访问令牌（Access Token）的方法

要获取使用应用程序的用户（以下简称“用户”）的访问令牌，请按照以下步骤请求发行。

:::tip

以下说明的方法称为 [OAuth 2.0](https://datatracker.ietf.org/doc/html/rfc6749.html)。通常的 OAuth 需要创建应用程序，但通过 [IndieAuth](https://indieauth.spec.indieweb.org/) 的扩展，现在无需创建应用即可使用。

针对 OAuth 方式有许多可用的库，建议尽可能直接使用现成的库。

目前若要使用此方式，您需要一个网页。如果无论如何都无法准备网页，或者您需要支持 Misskey 2023.9.0 之前的版本，请使用以下方式：

- [Misskey 专用的 MiAuth 方式获取访问令牌的方法](./miauth.md)
- [也有通过创建应用来获取访问令牌的方法（旧方法）](./app.md)

:::

## 步骤 1

创建一个用于介绍应用程序的网页。请确保可以通过 HTTPS 地址访问该页面。在页面的某处写入以下 HTML 代码：

```html
<!-- （必填项）href 的地址将作为认证代码的重定向（转发）目标。 -->
<link rel="redirect_uri" href="/redirect">

<div class="h-app">
  <!-- （选填 / v2025.3.0 起）可指定应用程序的标志（Logo）。推荐使用宽高比为 1:1 的标志。 -->
  <img src="/logo.png" class="u-logo">
  <!-- （选填）指定向用户展示的应用程序名称。若未指定，该页面的网址将作为默认名称。 -->
  <!-- （选填）href 属性必须设置为指向此页面（即此应用介绍页面的 URL）的链接。 -->
	<a href="/" class="u-url p-name">My Misskey App</a>
</div>
```

稍后，认证代码（Authorization Code）将被转发到 `redirect_uri` 所指定的地址。

---

从 Misskey v2026.3.0 开始，除了通过 HTML 进行的客户端发现之外，也开始支持 JSON 格式的客户端元数据。如果需要提供 JSON 格式的客户端元数据，请确保可以通过 HTTPS 地址访问如下所示的 JSON 文档。有关详细的规范，请参阅 [IndieAuth (IndieWeb Living Standard 11 July 2024)](https://indieauth.spec.indieweb.org/)。

另外需要注意的是，虽然此客户端发现与 OpenID Connect 的 Client Metadata 1.0 中定义的格式有一定程度的共通之处，但 Misskey 并不支持基于 OpenID Connect 的 OAuth，请留意这一点。

```json
{
  "client_id": "https://example.com", // 必须。客户端 ID 必须是存放此客户端元数据位置的 URL。
  "client_name": "My Misskey App", // 可选。指定向用户显示的应用程序名称。如果没有指定，客户端 ID 将作为名称。
  "client_uri": "https://example.com", // 必须。必须是 client_id 的前缀。
  "logo_uri": "https://example.com/logo.png", // 可选。可以指定应用程序的徽标。建议使用宽高比为 1:1 的徽标。
  "redirect_uris": [
    "https://example.com/callback" // 必须。指定授权码的转发目标 URL。
  ]
}
```

## 步骤 2

生成 PKCE 的 `code_verifier` 和 `code_challenge` 字符串，以及 `state` 字符串。

- `code_verifier`：长度限制为最少 43 个字符，最多 128 个字符，且仅限使用英文字母大小写以及 `-._~` 中的字符。
- `code_challenge`：将 `code_verifier` 字符串使用 SHA256 算法进行哈希处理后，再进行 base64url 编码的结果。
- `state`：没有特别的限制。请使用随机字符串。

:::danger

请每次都重新生成这些字符串，切勿重复使用。

:::

:::tip

建议使用像 [pkce-challenge](https://www.npmjs.com/package/pkce-challenge) 这样的库，或者直接使用 OAuth 库中自带的 PKCE 功能。

:::

:::tip{label='例'}

```js
import crypto from "node:crypto";

const chars = "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz-._~";
const codeVerifier = new Array(128)
  .fill(0)
  .map(() => chars[Math.floor(chars.length * Math.random())])
  .join("");
console.log('code_verifier', codeVerifier);

const codeChallenge = crypto
  .createHash("sha256")
  .update(codeVerifier, "ascii")
  .digest("base64url");
console.log('code_challenge', codeChallenge);

const state = crypto.randomUUID();
console.log('state', state);
```

:::

## 步骤 3

获取目标服务器的 OAuth 信息。数据为 JSON 格式。

```
https://{host}/.well-known/oauth-authorization-server
```

`{host}` 部分替换为用户服务器的主机名。通常主机名由用户输入。

在这里，我们会用到 `authorization_endpoint` 和 `token_endpoint`。

:::tip

在下一个步骤中会用到的 `scope` 信息，也可以通过 `scopes_supported` 字段进行确认。

:::

## 步骤 4

在用户的浏览器中显示应用程序认证表单。可以通过以下格式的 URL 打开认证表单：

```
{authorization_endpoint}?client_id={client_id}&response_type=code&redirect_uri={redirect_uri}&scope={scope}&code_challenge={code_challenge}&code_challenge_method=S256&state={state}
```

在此：

- `{authorization_endpoint}`：替换为在上一个步骤中获取的地址。
- `{client_id}`：替换为应用程序介绍页面的地址。
- `{code_challenge}`：替换为之前生成的 `code_challenge` 字符串。
- `code_challenge_method`：请始终设置为 `S256`。
- `{redirect_uri}`：替换为在介绍页面中设置的转发目标（重定向）地址。
- `{scope}`：替换为应用程序请求的权限。请将请求的权限以空格分隔列出。权限列表可以在[此处](../permission.md)确认。
- `{state}`：替换为之前生成的 state 字符串。

:::tip{label='例'}

```
https://misskey.local/oauth/authorize?client_id=http%3A%2F%2Fexample.com&code_challenge=C6hwMO2bmIzg3nqppTE9b79fvuOjlrKmH2xNiZSMHzw&code_challenge_method=S256&response_type=code&redirect_uri=http%3A%2F%2Fexample.com%2Fredirect&scope=write%3Anotes&state=87c11f05-86eb-4eb2-9057-f6a98fc5e9ab
```

:::

## 步骤 5

当用户允许应用程序访问后，认证代码将以 URL 参数的形式被转发到 `redirect_uri` 的地址。

| 名称      | 说明                    |
| ------- | --------------------- |
| `code`  | 用户的认证代码。              |
| `state` | 认证请求中使用的 `state` 字符串。 |

:::tip{label='例'}

```
https://example.com/redirect?code=...&state=87c11f05-86eb-4eb2-9057-f6a98fc5e9ab
```

:::

请确认收到的 `state` 字符串是否与当初发送的一致，确认无误后再进入下一个步骤。

## 步骤 6

使用转发过来的认证代码，通过 POST 方法请求访问令牌。请求目标为 `token_endpoint`。数据格式支持 `application/json` 和 `application/x-www-form-urlencoded`。各项参数设定如下：

| 名称              | 说明                           |
| --------------- | ---------------------------- |
| `grant_type`    | 始终设置为 `authorization_code`。  |
| `client_id`     | 认证请求中使用的 `client_id` 字符串。    |
| `redirect_uri`  | 认证请求中使用的 `redirect_uri` 字符串。 |
| `scope`         | 认证请求中使用的 `scope` 字符串。        |
| `code`          | 获取的认证代码。                     |
| `code_verifier` | 之前生成的 `code_verifier` 字符串。   |

:::tip{label='例'}

```js
const res = await fetch(endpoint, {
  method: "POST",
  body: JSON.stringify({
    grant_type: "authorization_code",
    client_id: "https://example.com",
    redirect_uri: "https://example.com/redirect",
    scope: "write:notes",
    code: "...",
    code_verifier: "hjjbCYDmDpSLjirkO-PrfWKsRhDdJr-PAEGRClRwzUKlmFIIIrZNmSvUIraeIa~WqbqQnfbJV-Hc_IfuQkesBYUpukUi~lInDfU_AZjoZqbU.ioQTRzaFfZFfGnT-OAA",
  }),
  headers: {
    "Content-Type": "application/json"
  }
});
```

:::

响应内容将为 JSON 对象格式，您可以从该对象中获取并使用 `access_token`。
